# Asset Tag Import

This is the repeatable pipeline that turns Dungeondraft `.dungeondraft_tags`
files into the final `data/assets/asset-manifest.json`. See
[ASSET_TAXONOMY.md](ASSET_TAXONOMY.md) for the data model and query API.

## Importing new `.dungeondraft_tags`

Drop new packs anywhere under `local-assets/`, `assets/`, or `data/raw-assets/`
(each pack keeps its `data/*.dungeondraft_tags` and its `textures/` folder), then
re-run the pipeline. The importer auto-discovers every `*.dungeondraft_tags`
file recursively.

You can also point the importer at an explicit path or glob:

```bash
pnpm assets:import-tags -- "local-assets/**/*.dungeondraft_tags"
pnpm assets:import-tags -- "local-assets/CH - Sea/data/default.dungeondraft_tags"
```

If no raw tag files are found, the importer falls back to the pre-merged v3
export (`data/assets/imports/dm_instamap_merged_dungeondraft_tags_v3.json`) so a
fresh checkout still works.

The parser is **permissive**: it tolerates trailing commas, deduplicates by
`sourceTag + path`, and preserves the original sourceTags per asset.

## Commands to run (in order)

```bash
pnpm assets:import-tags     # -> data/assets/imports/imported-tags.json
pnpm assets:map-taxonomy    # -> mapped-assets.json + taxonomy/asset-taxonomy.json
pnpm assets:metadata        # -> mapped-assets.with-metadata.json  (sharp; slow)
pnpm assets:audit-manifest  # -> reports/audit-report.json + .md
pnpm assets:manifest        # -> data/assets/asset-manifest.json (FINAL) + asset-groups.json
pnpm assets:validate        # exits non-zero on hard failures
pnpm assets:contact-sheets -- --only unknown,needs-review,suspicious-light
```

Notes:

- `assets:metadata` reads every image with `sharp`; use `-- --limit N` for a
  quick partial run during development.
- `assets:map-taxonomy` uses the v3 per-tag taxonomy
  (`data/assets/taxonomy/dm_instamap_taxonomy_merged_v3.json`) as the
  authoritative **seed**; tags not in the seed fall back to the keyword rule
  engine in [`mapping.ts`](../packages/assets/src/taxonomy/mapping.ts).
- `assets:audit-manifest` is the new pipeline audit. The legacy
  `pnpm assets:audit` (scanner manifest under `data/indexes/`) is unchanged.
- `assets:manifest` also (re)generates `data/indexes/asset-groups.json` from the
  taxonomy — a few hundred semantic `macroCategory / assetGroup` groups that the
  exporter resolver, editor and AI bridge consume. The old folder/aspect/colour
  grouping is gone; `pnpm assets:group` now just regenerates that index from the
  existing manifest (no rescan needed).

## Reviewing groups (`/asset-groups/review`)

The web review at `/asset-groups/review` works on the semantic taxonomy groups.
For a group you can **approve** it, **set a status**
(approved / needs-review / quarantine / rejected), or **correct** its
`macroCategory` / `assetGroups` / `themeTags`. Each action is written as
asset-level overrides (keyed by path) into
`data/assets/overrides/asset-overrides.json`. Re-run `pnpm assets:manifest` to
apply them and refresh the manifest + groups index.

## Where the generated files go

```
data/assets/
  imports/
    imported-tags.json                 # sourceTag -> paths, asset -> sourceTags
    dm_instamap_merged_dungeondraft_tags_v3.json   # provided v3 input
  taxonomy/
    asset-taxonomy.json                # per sourceTag normalized mapping
    dm_instamap_taxonomy_merged_v3.json            # provided v3 seed
  manifests/
    mapped-assets.json                 # after taxonomy mapping
    mapped-assets.with-metadata.json   # after metadata enrichment
  reports/
    audit-report.json / audit-report.md
  overrides/
    asset-overrides.json               # manual overrides (tracked)
  contact-sheets/<bucket>/page-001.png # anomaly thumbnails
  asset-manifest.json                  # FINAL — consumed by the generator
```

> The large generated JSON files (`mapped-assets*.json`, `asset-manifest.json`,
> the v3 inputs) are big (tens of MB). They are regenerated by this pipeline, so
> avoid committing them; keep `overrides/asset-overrides.json` and the small
> reports in git.

## Reading the audit report

`audit-report.md` lists each check with a count and the first examples:

- **File mancanti sul disco** — manifest path not found under the asset roots.
- **macroCategory unknown** — kept as `needs-review`; needs an override/rule.
- **conflitto di categoria** — multiple sourceTags voted for different macros.
- **suspicious-light / carpet-as-light** — must be `0`; otherwise the anti-light
  rules regressed and `assets:validate` will fail.
- **molto piccoli / senza trasparenza** — quality signals for manual review.
- **VM non marcato** — a path contains `VM` but `sourcePacks` is missing it.

Generate visual contact sheets for the anomaly buckets to eyeball them:

```bash
pnpm assets:contact-sheets -- --only unknown,needs-review,suspicious-light
# each thumbnail shows id, macroCategory, assetGroups, leading sourceTags
```

## Adding manual overrides

Edit `data/assets/overrides/asset-overrides.json`. Overrides are applied **after**
automatic mapping and **before** validation, in increasing specificity:
`packs` -> `groups` (by sourceTag) -> `assets` (by exact path).

```jsonc
{
  "assets": {
    "textures/objects/red_carpet_01.png": {
      "macroCategory": "decoration",
      "assetGroups": ["carpet", "runner"],   // asset-level lists REPLACE
      "themeTags": ["tavern", "noble"],
      "usageRules": { "canBeLightEmitter": false, "canBeFloorOverlay": true },
      "status": "approved"
    }
  },
  "groups": {
    ".Table": {                                // group-level lists UNION
      "macroCategory": "furniture",
      "themeTags": ["tavern", "interior", "wood"],
      "placementTags": ["room_center", "near_chairs"]
    }
  },
  "packs": {
    "VM": { "preserveAsSourcePack": true, "stripFromNormalizedGroups": true }
  }
}
```

After editing overrides, re-run `pnpm assets:manifest` and `pnpm assets:validate`.

## Regenerating the final manifest

The minimal loop after any change (new pack, rule tweak, override):

```bash
pnpm assets:import-tags && pnpm assets:map-taxonomy && pnpm assets:manifest && pnpm assets:validate
```

Run `pnpm assets:metadata` before `assets:manifest` whenever you need fresh image
metadata (dimensions/transparency/hash). The generator reads only the final
`data/assets/asset-manifest.json`.
